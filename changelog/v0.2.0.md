# v0.2.0 - Job lifecycle management

Release date: 2026-06-13

## Summary

This release introduces a complete job lifecycle for the broker.

Jobs are now assigned unique identifiers, transition into a processing state when dequeued, can be acknowledged on success, retried on failure, and are automatically moved to a dead-letter queue after exhausting the maximum retry limit.

## Added

### Job IDs

Every enqueued job is now assigned a unique identifier.

Example:

```
job-1
job-2
job-3
```

### Processing State

Dequeued jobs are no longer removed immediately. They enter a processing state until they are acknowledged or failed.

### Acknowledgement

Added support for:

```
ack <job_id>
```

which permanently removes a processing job from the broker.

### Failure Handling

Added support for:

```
fail <job_id>
```

which marks a processing job as failed.

### Retry Tracking

Jobs now maintain:

- `attempts`
- `max_attempts`

Each failure increments the attempt counter before the job is requeued.

### Dead Letter Queue

Jobs that reach their maximum retry count are automatically moved into a dead-letter queue instead of being requeued indefinitely.

### Enhanced Queue Listing

The `list` command now displays three broker states:

- Queued
- Processing
- Dead Letter

including the current retry count for each job.

## Changed

- Jobs are now represented by a structured object instead of a payload string.
- Dequeue now moves jobs into processing instead of removing them completely.
- Failed jobs are requeued to the back of the queue.

## Notes

The broker remains entirely in-memory.

Queued jobs, processing jobs, and dead-letter jobs are not persisted after the application exits.
