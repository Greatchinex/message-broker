# v0.3.0 - Durable Event Log & Recovery

Release date: 2026-06-13

## Summary

This release introduces durability to the message broker through an append-only event log.

Every state-changing operation is now recorded to disk, allowing the broker to reconstruct its state when restarted. Instead of serializing the entire broker, the system rebuilds itself by replaying the recorded events in order.

This lays the foundation for reliable message processing and future persistence improvements.

## Added

### Append-Only Event Log

All mutating broker operations are now written to a log file before being applied to in-memory state.

Logged events include:

- Enqueue
- Dequeue
- Acknowledge
- Fail
- Dead-letter transitions

### Broker Recovery

On startup, the broker reads the event log and replays every recorded operation to rebuild its internal state.

No manual intervention is required to restore the queue after a restart.

### Event Replay

Broker state is reconstructed by replaying historical events instead of loading a serialized snapshot.

This approach keeps the persistence layer simple while mirroring techniques used by many production systems.

## Changed

- State-changing commands now follow a write-then-apply workflow.
- Broker initialization now performs recovery before accepting new commands.

## Notes

The broker currently relies entirely on event replay for recovery.

Log compaction and snapshotting are not yet implemented, so startup time will grow as the event log increases.

## Example

```text
enqueue send_email
enqueue generate_invoice
dequeue
ack job-1
```

After restarting the application, replaying these events restores the broker to the same state it had before shutdown.
